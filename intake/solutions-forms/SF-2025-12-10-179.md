# Solution Request #179 - Birla Novel
**Date:** 2025-12-10  |  **Status:** Solution Given Closed  |  **Complexity:** Not Set

## Subject
Enhancement Request: Award-Specific Approvers in Dashboard

## Brief
Assign Award Specific Approver from Dashboard for Standard Awards

## Problem Description
Could we explore the possibility of adding Award-Specific Approvers in the Dashboard, similar to the existing “Assign Nominator” feature, with support for at least up to a second-level approval?

The reason for this request is that many clients who use HRIS for data integration are reluctant to input approver details in the additional info fields when there is no consistency in those details (e.g., no defined designation, role, or structure).

Could you kindly help us review this enhancement request? We have received similar feedback from multiple clients, and hardcoding approver details is not a sustainable solution.

## Solution Given
After validating the requirement for assigning an approver at the award level, please find below our observations.
Feasibility
Assigning a constant approver to an award from the admin dashboard is feasible for 1st and 2nd level approvals.
It is not feasible for 3rd and 4th level approvals, as these levels are configured from the backend at the code level.
However, enabling this capability comes with certain implications that may create friction in real-life scenarios. These are outlined below.
Implications
A single user can be mapped as an approver for either 1st or 2nd level, and all nominations will flow to that user at the respective level, irrespective of hierarchy or relevance mapping, which clients typically expect.
For example, clients often require 1st level approval to go to the nominator’s RM and 2nd level approval to the Department Head.
Such relevance-based routing will not be possible if only a constant user is mapped as the 1st or 2nd level approver.
If user A is mapped as the 1st level approver, all nominations will go to user A regardless of the nominator/department/nominee relationship.
If budget deduction occurs at the approver level, the user assigned as the 1st or 2nd level approver must have budget allocated for all relevant countries. Otherwise, nominations may remain pending due to non-availability of budget.
Before assigning a user as the 1st or 2nd level approver, the user must be granted Approver rights. Without these rights, nominations will flow to the user, but they will not appear in their To-Do section for action.
There may be scenarios where an assigned approver ends up approving their own nomination(s).
For instance, if an award has 2 levels of approval and user A is mapped as the 2nd level approver, user C might nominate user A. In that case, the 2nd level approval will route to user A.
Clients may raise concerns upon discovering such cases, potentially leading to additional discussion, validation, and client expectation management.
Considering all the above implications, please share your thoughts on whether this capability should be added to the dashboard, as the use of constant approvers may introduce challenges from an ongoing client management perspective.
